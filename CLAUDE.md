# Real Estate Listing Video Generator

## 1. Project Overview

This project turns a realtor's listing photos and property details into a finished, branded marketing video automatically — no video editing software, no manual cutting.

The realtor uploads 4 photos and fills in property details once. Within a couple of minutes the video is ready to download from the same frontend page: it opens with the agent's own photo, name, phone, email, and socials for the first 5 seconds, then moves through the 4 property photos (each with an animated Ken Burns zoom/pan and cross-dissolve transitions), a lower third showing price, location, and key features, and looping background music underneath.

**Why Creatomate, not Higgsfield/Runway/Kling:** those tools use generative AI to animate a still image. Powerful, but for a first build they mean two chained APIs, async job polling, retries, and a separate compositing step just for text overlays and stitching. Creatomate does the animation (Ken Burns), lower thirds, and final assembly in one tool and one template — dramatically simpler, cheaper, and more reliable to ship on a deadline. AI-animated motion is an explicit future upgrade once this baseline is proven, not part of this build.

## 2. The WAT Framework

This project is organized around **Workflows, Agent, Tools**:

- **W — Workflows** (`/workflows/`): step-by-step procedure files that describe *how* the pipeline runs — the 5-step process below. These are the source of truth for what should happen; Claude Code reads them before acting.
- **A — Agent**: Claude Code itself. No folder — it reads the workflow files, plans the work, and executes it using the Tools. When you ask for a change, expect the agent to check `/workflows/` first for the relevant procedure before touching anything.
- **T — Tools** (`/tools/`): scripts and integrations the agent calls to actually do things — n8n workflow JSON exports, Creatomate template helpers, Google Sheets helpers, deployment scripts. Tools should stay thin wrappers around n8n/Creatomate/Sheets, not reimplementations of logic that already lives in those systems.

## 3. The Pipeline (5 steps, one n8n workflow)

1. **Realtor intake** — n8n Form Trigger (or the Vercel frontend, see §6) collects 4 photos, price, location, address, bed/bath, key features, description, and the agent's email.
2. **Agent profile lookup** — Google Sheets node reads the agent roster tab, matching by the submitter's email, so agents never re-enter their own headshot/name/phone/socials.
3. **Send to Creatomate** — HTTP Request node fills the Creatomate template's placeholders (`{{price}}`, `{{agent_name}}`, `{{photo_1}}`, etc.) and triggers a render. The template itself (5 scenes, lower thirds, zoom/pan keyframes, transitions, looping music track) is built once in Creatomate's visual editor — it is not generated by code.
4. **Poll for completion** — Wait node + HTTP status check loop until the render is done (standard async pattern, not custom-built).
5. **Deliver** — n8n writes the finished video URL to the Jobs sheet row (see §5); the frontend, which has been polling that row, shows a **Download** button as soon as the URL appears. No email or Slack step — the frontend is the delivery mechanism.

## 5. Tech Stack

- **n8n** — orchestrates the 5-step pipeline (form trigger, Sheets lookup, HTTP calls to Creatomate, polling, delivery).
- **Google Sheets** — two tabs in one spreadsheet:
  - *Agents* tab: one row per agent (headshot, name, phone, email, socials), read-only lookup.
  - *Jobs* tab: one row per submission (`submissionId`, `status`, `videoUrl`, timestamps), written and updated by n8n as each render progresses.
- **Creatomate** — the render engine: Ken Burns zoom/pan, lower thirds, transitions, background music, final video assembly. One template, built visually, reused per render.
- **Vercel** — hosts the frontend intake form (`/frontend/`), independent deploys from the n8n workflow.

**Alternatives considered and deferred:** Airtable or a small database (e.g. Supabase) instead of Google Sheets for job status. At current expected volume, a single spreadsheet with two tabs is simpler and sufficient — revisit only if concurrent renders make Sheets' write concurrency a real bottleneck.

## 6. Architecture: How the Frontend and Backend Talk to Each Other

```
Vercel (frontend)                          n8n (backend)
─────────────────                          ─────────────
1. POST intake data/photos  ───────────▶   Intake webhook
                                            → generates submissionId
   ◀─────────────────────────────────────  returns { submissionId }

2. n8n runs the pipeline (§3), writing/updating a row in the
   Jobs sheet tab as it goes: submitted → rendering → complete/error

3. Frontend polls, using submissionId  ───▶ Status webhook
   every few seconds                         → reads matching Jobs row
   ◀─────────────────────────────────────  returns { status, videoUrl? }

4. Once status is "complete", the frontend shows a Download button
   using the returned videoUrl. This is the only delivery path —
   no email or Slack notification.
```

- **Two n8n webhooks**: an intake webhook (starts the pipeline, returns a `submissionId` immediately) and a status webhook (reads the Jobs sheet row for a given `submissionId` and returns JSON). Both URLs are stored as Vercel environment variables (`N8N_INTAKE_WEBHOOK_URL`, `N8N_STATUS_WEBHOOK_URL`) — never hardcoded in frontend code.
- **The Jobs sheet tab is the shared state** between the two systems — n8n writes it, the status webhook reads it. No separate database needed for v1.
- **Delivery is frontend-only**: the realtor gets the video by staying on the submission page and clicking Download once it's ready — there's no email/Slack fallback. Since there's no way to recover the video if they navigate away, the frontend shows a persistent "Don't close this window — your video is being generated" message for the whole pending/rendering state.

## 7. Frontend (`/frontend/`)

**What it needs to do:**
- Collect the same intake fields as the n8n form: 4 photo uploads, price, location, address, bed/bath, key features, description, and the agent's email (used for the Sheets lookup).
- On submit, POST to the n8n intake webhook and receive a `submissionId`.
- Poll the n8n status webhook with that `submissionId`. While pending/rendering, show a persistent "Don't close this window — your video is being generated" message. Once complete, show a **Download** button (linking to the video URL). This is the only delivery mechanism — no email or Slack notification.
- Nothing else — no business logic, validation beyond basic form checks, or video processing happens client-side; that all stays in n8n + Creatomate.

**Where it lives:** `/frontend/` — a Next.js/React app, deployed to Vercel independently of the n8n workflow. Built with the `frontend-design` skill (installed separately) for the visual/UX work.

**Conventions for the agent when generating or editing frontend code:**
- Keep `N8N_INTAKE_WEBHOOK_URL` and `N8N_STATUS_WEBHOOK_URL` in Vercel environment variables — never hardcode a webhook URL in committed code.
- Prefer routing webhook calls through Next.js route handlers/server actions rather than calling n8n directly from client-side JS, so the webhook URLs aren't exposed in the browser bundle.
- Don't duplicate pipeline logic (render triggering, status computation) in the frontend — it should only submit, poll, and display.
- Use Vercel preview deployments to test form/UI changes before promoting to production.

## 8. Conventions for the Agent

- Never commit `.env`, or anything under `/temp/outputs/` or `/temp/resources/` — these are gitignored scratch/output space, not deliverables.
- n8n workflow JSON exports live in `/workflows/`. Once the Creatomate-only baseline is tested and working, tag/branch it (e.g. `v1-ken-burns`) before any future AI-animation upgrade is merged in.
- Treat the Creatomate template and the n8n workflow as the source of truth for pipeline behavior — `/tools/` scripts should be thin wrappers around them, not reimplementations.
- Check `/workflows/` for the relevant procedure file before making pipeline changes; update the procedure file if the actual workflow changes.
- This is a foundation doc for a small, deadline-driven build — keep additions simple and avoid introducing new services/tools without flagging the tradeoff first.

# Real Estate Listing Video Generator

## 1. Project Overview

This project turns a realtor's listing photos and property details into a finished, branded marketing video automatically — no video editing software, no manual cutting.

The realtor uploads 4 property photos plus their own photo, and fills in property details once. Within a couple of minutes the video is ready to download from the same frontend page: it opens with the 4 property photos (each with an animated Ken Burns zoom/pan and cross-dissolve transitions), a lower third showing price, address, and bed/bath/sqft/year-built details, then closes with the realtor's uploaded photo, name, phone, and email. Looping background music plays underneath the full video.

**Why Creatomate, not Higgsfield/Runway/Kling:** those tools use generative AI to animate a still image. Powerful, but for a first build they mean two chained APIs, async job polling, retries, and a separate compositing step just for text overlays and stitching. Creatomate does the animation (Ken Burns), lower thirds, and final assembly in one tool and one template — dramatically simpler, cheaper, and more reliable to ship on a deadline. AI-animated motion is an explicit future upgrade once this baseline is proven, not part of this build.

## 2. The WAT Framework

This project is organized around **Workflows, Agent, Tools**:

- **W — Workflows** (`/workflows/`): step-by-step procedure files that describe *how* the pipeline runs — the process below. These are the source of truth for what should happen; Claude Code reads them before acting.
- **A — Agent**: Claude Code itself. No folder — it reads the workflow files, plans the work, and executes it using the Tools. When you ask for a change, expect the agent to check `/workflows/` first for the relevant procedure before touching anything.
- **T — Tools** (`/tools/`): scripts and integrations the agent calls to actually do things — n8n workflow JSON exports, Creatomate template helpers, Google Sheets helpers, deployment scripts. Tools should stay thin wrappers around n8n/Creatomate/Sheets, not reimplementations of logic that already lives in those systems.

## 3. The Pipeline (n8n workflow: "Listing Video Intake, Render & Poll")

1. **Realtor intake** — a webhook (the future Vercel frontend, see §6) collects 4 property photos, the realtor's own photo, price, address, bed/bath/sqft/year built, key features, description, and the agent's email. Responds immediately with a `submissionId`.
2. **Agent profile lookup** — Google Sheets node reads the Agents roster tab, matching by the submitter's email, so agents never re-enter their own name/phone/socials. (The realtor's photo now comes from the frontend upload, not this lookup — see §5.) If no match, the job is marked `agent_not_found` and stops there.
3. **Send to Creatomate** — HTTP Request node fills the Creatomate template's real element names (`Picture-01`/`Picture-02`/`Picture-03`/`Picture-04`, `Realtor`, `Address`, `Details`, `Price`, `Description`, `Realtor-Name`, `Realtor-Phone`, `Realtor-Email`) and triggers a render. The template itself (built once in Creatomate's visual editor, not generated by code) is Project `Realtor-video-generator` (id `9050cc5b-154e-494a-b2a9-200833d8ae2e`), Template `48be3d34-9888-481c-8ebe-01778a49e255`. There's also a static `Title` element ("NEW LISTING") that isn't sent per-render — it's fixed template text. The template has no slot for the realtor's website/social handle; that field is currently dropped, not rendered anywhere.
   - **Resolved quirk — `Address`/`Price` element names in this template**: earlier, the element *named* `Address` was visually styled/positioned as the price display and the element *named* `Price` was visually the address display, so the HTTP node compensated by sending the price value under the `Address` key and the address value under the `Price` key (fixed 2026-07-23). On 2026-07-23, the realtor renamed the two Creatomate layers directly so each name now matches what it visually displays — `Address` shows the address, `Price` shows the price. The n8n HTTP node was updated to match (price → `Price` key, address → `Address` key), removing the old compensation. If price/address ever look swapped again in a render, check whether the Creatomate layer names have changed before touching the n8n node — the source of truth is now "does the key name match what the layer visually shows," not a baked-in swap.
4. **Poll for completion** — Wait node + HTTP status check loop until the render is done, capped at ~40 attempts (~10 min) before marking the job `error`.
5. **Deliver** — n8n writes the finished video URL to the `ListingVideoJobs` Data Table row (see §5); the frontend, which has been polling that row via the status webhook, shows a **Download** button as soon as the URL appears. This is the primary delivery mechanism.
6. **Backup email** — once the Data Table row is marked `complete`, a Gmail node emails the matched agent (from the Sheets lookup, step 2) a download link for the video. This exists only as a fallback in case the realtor navigates away before the frontend shows the Download button — the frontend page is still the primary/expected delivery path. No email is sent on `error` or `agent_not_found`.

## 5. Tech Stack

- **n8n** — orchestrates the pipeline (intake webhook, Sheets lookup, HTTP calls to Creatomate, polling, status webhook, backup email). Two workflows: `Listing Video Intake, Render & Poll` and `Listing Video Status Check`.
- **Google Sheets** — Agents-roster only: spreadsheet "Real_Estate_Video_Generator" (id `1akzdENiWG3D4zu4zSLUv4B4xLrA98GO8-oDr1tA4uc4`), tab "Agents", one row per agent (`Email`, `Name`, `Phone`, `HeadshotUrl`, `Website`, `Instagram`), read-only lookup. `HeadshotUrl` is no longer used by the render (the realtor's photo now comes from a frontend upload instead) — the column can stay in the sheet unused or be removed, it doesn't matter.
- **n8n Data Table** — `ListingVideoJobs` (id `2j4dIhU0LEFKBFA8`): one row per submission (`submissionId`, `status`, `videoUrl`, `creatomateRenderId`, `agentEmail`, `errorMessage`, `pollAttempts`, `rawIntakePayload`), written and updated by n8n as each render progresses. Chosen over a second Google Sheets tab because it's written/updated multiple times per render and a native Data Table avoids Sheets' write-concurrency/rate-limit quirks.
- **Creatomate** — the render engine: Ken Burns zoom/pan, lower thirds, transitions, background music, final video assembly. One template, built visually, reused per render.
- **Gmail** (`gmailOAuth2` credential, "Gmail account") — sends the backup delivery email on render completion (see §3 step 6). OAuth2 credentials expire periodically; if the "Send Video Email" node starts failing with a "needs to be reconnected" error, reconnect it from n8n's Credentials list — this can't be done from the agent, it requires the browser OAuth flow.
- **Vercel** — hosts the frontend intake form (`/frontend/`), independent deploys from the n8n workflow.
- **Vercel Blob store access mode matters and can't be changed later**: when creating the Blob store, you must explicitly choose **Public** access — the dashboard defaults to and recommends **Private**, but a private store rejects the `access: 'public'` uploads this app needs (Creatomate has to fetch photo URLs directly, with no auth). Access mode is locked in at store creation; if a store ends up Private, the fix is deleting it and creating a new Public one, not a settings toggle.

## 6. Architecture: How the Frontend and Backend Talk to Each Other

```
Vercel (frontend)                          n8n (backend)
─────────────────                          ─────────────
1. POST intake data/photos  ───────────▶   Intake webhook
                                            → generates submissionId
   ◀─────────────────────────────────────  returns { submissionId }

2. n8n runs the pipeline (§3), writing/updating a row in the
   ListingVideoJobs Data Table as it goes: submitted → rendering → complete/error

3. Frontend polls, using submissionId  ───▶ Status webhook
   every few seconds                         → reads matching Data Table row
   ◀─────────────────────────────────────  returns { status, videoUrl? }

4. Once status is "complete", the frontend shows a Download button
   using the returned videoUrl — the primary delivery path. n8n also
   emails the agent a backup download link at the same point (§3 step 6).
```

- **Two n8n webhooks/workflows**: `Listing Video Intake, Render & Poll` (starts the pipeline, returns a `submissionId` immediately) and `Listing Video Status Check` (reads the `ListingVideoJobs` Data Table row for a given `submissionId` and returns JSON). Both webhook URLs are stored as Vercel environment variables (`N8N_INTAKE_WEBHOOK_URL`, `N8N_STATUS_WEBHOOK_URL`) — never hardcoded in frontend code.
- **The `ListingVideoJobs` Data Table is the shared state** between the two systems — the intake workflow writes it, the status workflow reads it.
- **Delivery is frontend-first, email-backup**: the realtor gets the video by staying on the submission page and clicking Download once it's ready; a backup email with the same link goes out in parallel in case they navigate away. The frontend still shows a persistent "Don't close this window — your video is being generated" message for the whole pending/rendering state, since the email is a fallback, not the primary UX.

## 7. Frontend (`/frontend/`)

**What it needs to do:**
- Collect the same intake fields the n8n workflow expects: 4 property photo uploads, **the realtor's own photo upload**, price, address, bed/bath, sqft, year built, key features, description, and the agent's email (used for the Sheets lookup — name/phone/socials only, not the photo). Description is wired into the Creatomate render (`Description` element); key features remains frontend-only — the template has no slot for it.
- On submit, POST to the n8n intake webhook and receive a `submissionId`.
- Poll the n8n status webhook with that `submissionId`. While pending/rendering, show a persistent "Don't close this window — your video is being generated" message. Once complete, show a **Download** button (linking to the video URL). This is the primary delivery mechanism; n8n also sends a backup email with the same link (see §3, §6) in case the realtor navigates away.
- Nothing else — no business logic, validation beyond basic form checks, or video processing happens client-side; that all stays in n8n + Creatomate.

**Where it lives:** `/frontend/` — a Next.js (App Router, TypeScript, Tailwind v4) app, deployed to Vercel independently of the n8n workflow. Built with the `frontend-design` skill for the visual/UX work — see §8 for the design direction.

**Photo uploads → Vercel Blob, not through the server:** the 5 photo files (4 property + realtor) are uploaded **directly from the browser** to Vercel Blob via `upload()` from `@vercel/blob/client`, authorized by a short-lived token minted by `app/api/blob-upload/route.ts`. This is deliberate, not incidental — routing real phone-camera photos (3–10MB each) through a Server Action or Route Handler risks Vercel's serverless body-size ceiling (Server Actions cap at 1MB by default; general function limits are ~4.5MB). Client-direct upload means file bytes never pass through a Next.js server function at all. Only after all 5 uploads resolve does the client call `submitIntake()` (a Server Action) with the resulting URLs.

**Conventions for the agent when generating or editing frontend code:**
- Keep `N8N_INTAKE_WEBHOOK_URL`, `N8N_STATUS_WEBHOOK_URL`, and `BLOB_READ_WRITE_TOKEN` in environment variables (`frontend/.env.local` locally, Vercel project settings in deployment) — never hardcode a webhook URL or token in committed code. `frontend/.env.example` documents all three.
- Prefer routing webhook calls through Next.js Route Handlers/Server Actions rather than calling n8n directly from client-side JS, so the webhook URLs aren't exposed in the browser bundle. The one exception is the Blob upload itself, which is intentionally client-direct (see above) — it doesn't touch the n8n webhooks.
- Don't duplicate pipeline logic (render triggering, status computation) in the frontend — it should only submit, poll, and display.
- The status proxy (`app/api/status/route.ts`) treats a 404 from the n8n status webhook as still-pending, not an error — the intake workflow responds with a `submissionId` in parallel with (not gated on) its Data Table row write, so a client can legitimately poll before that row exists.
- Use Vercel preview deployments to test form/UI changes before promoting to production.

## 8. Frontend Design Direction

The intake form and status page are deliberately styled as a two-act experience rather than a generic SaaS form:

- **Act I — the intake slip** (`/`): warm kraft-paper palette (`--paper`, `--paper-card`, `--ink`, `--stamp` red, `--brass`), a heavy poster slab serif (Alfa Slab One) for headlines, and a typewriter face (Special Elite) for labels/badges — evokes a real estate office's paper intake ticket, not an editorial cream-and-serif template. Signature element: a rotated rubber-stamp badge ("Listing Intake · File No. ####") that settles into place on load.
- **Act II — the screening room** (`/status/[submissionId]`): dark near-black palette with an amber "projector" accent, same typewriter face for the status readout. Signature element: a 12-tick film-leader countdown ring that glows while pending, replaced by a "Now Showing" panel with a one-time light-sweep animation on completion.
- Both scenes respect `prefers-reduced-motion` and have visible keyboard focus rings in their own accent color.
- Design tokens live in `frontend/src/app/globals.css` (`@theme inline` block); fonts are loaded via `next/font/google` in `frontend/src/app/layout.tsx`.

## 10. Conventions for the Agent

- Never commit `.env`, or anything under `/temp/outputs/` or `/temp/resources/` — these are gitignored scratch/output space, not deliverables.
- n8n workflow JSON exports live in `/workflows/`. Once the Creatomate-only baseline is tested and working, tag/branch it (e.g. `v1-ken-burns`) before any future AI-animation upgrade is merged in.
- Treat the Creatomate template and the n8n workflow as the source of truth for pipeline behavior — `/tools/` scripts should be thin wrappers around them, not reimplementations.
- Check `/workflows/` for the relevant procedure file before making pipeline changes; update the procedure file if the actual workflow changes.
- This is a foundation doc for a small, deadline-driven build — keep additions simple and avoid introducing new services/tools without flagging the tradeoff first.

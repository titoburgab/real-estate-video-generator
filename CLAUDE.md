# Real Estate Listing Video Generator

## 1. Project Overview

This project turns a realtor's listing photos and property details into a finished, branded marketing video automatically вЂ” no video editing software, no manual cutting.

The realtor uploads 3 property photos plus their own photo, and fills in property details once. Within a couple of minutes the video is ready to download from the same frontend page: it opens with the 3 property photos (each with an animated Ken Burns zoom/pan and cross-dissolve transitions), a lower third showing price, address, and bed/bath/sqft/year-built details, then closes with the realtor's uploaded photo, name, phone, and website/social handle. Looping background music plays underneath the full video.

**Why Creatomate, not Higgsfield/Runway/Kling:** those tools use generative AI to animate a still image. Powerful, but for a first build they mean two chained APIs, async job polling, retries, and a separate compositing step just for text overlays and stitching. Creatomate does the animation (Ken Burns), lower thirds, and final assembly in one tool and one template вЂ” dramatically simpler, cheaper, and more reliable to ship on a deadline. AI-animated motion is an explicit future upgrade once this baseline is proven, not part of this build.

## 2. The WAT Framework

This project is organized around **Workflows, Agent, Tools**:

- **W вЂ” Workflows** (`/workflows/`): step-by-step procedure files that describe *how* the pipeline runs вЂ” the process below. These are the source of truth for what should happen; Claude Code reads them before acting.
- **A вЂ” Agent**: Claude Code itself. No folder вЂ” it reads the workflow files, plans the work, and executes it using the Tools. When you ask for a change, expect the agent to check `/workflows/` first for the relevant procedure before touching anything.
- **T вЂ” Tools** (`/tools/`): scripts and integrations the agent calls to actually do things вЂ” n8n workflow JSON exports, Creatomate template helpers, Google Sheets helpers, deployment scripts. Tools should stay thin wrappers around n8n/Creatomate/Sheets, not reimplementations of logic that already lives in those systems.

## 3. The Pipeline (n8n workflow: "Listing Video Intake, Render & Poll")

1. **Realtor intake** вЂ” a webhook (the future Vercel frontend, see В§6) collects 3 property photos, the realtor's own photo, price, address, bed/bath/sqft/year built, key features, description, and the agent's email. Responds immediately with a `submissionId`.
2. **Agent profile lookup** вЂ” Google Sheets node reads the Agents roster tab, matching by the submitter's email, so agents never re-enter their own name/phone/socials. (The realtor's photo now comes from the frontend upload, not this lookup вЂ” see В§5.) If no match, the job is marked `agent_not_found` and stops there.
3. **Send to Creatomate** вЂ” HTTP Request node fills the Creatomate template's real element names (`Photo-1`/`Photo-2`/`Photo-3`, `Realtor-Photo`, `Address`, `Details`, `Price`, `CTA-Text`, `Realtor-Name`, `Realtor-Phone`, `Realtor's-email`) and triggers a render. The template itself (built once in Creatomate's visual editor, not generated by code) is Project `Real_Estate_Video_Generator`, Template `c14aac66-5d48-4db4-946f-2fd792b44ff3`.
4. **Poll for completion** вЂ” Wait node + HTTP status check loop until the render is done, capped at ~40 attempts (~10 min) before marking the job `error`.
5. **Deliver** вЂ” n8n writes the finished video URL to the `ListingVideoJobs` Data Table row (see В§5); the frontend, which has been polling that row via the status webhook, shows a **Download** button as soon as the URL appears. No email or Slack step вЂ” the frontend is the delivery mechanism.

## 5. Tech Stack

- **n8n** вЂ” orchestrates the pipeline (intake webhook, Sheets lookup, HTTP calls to Creatomate, polling, status webhook). Two workflows: `Listing Video Intake, Render & Poll` and `Listing Video Status Check`.
- **Google Sheets** вЂ” Agents-roster only: spreadsheet "Real_Estate_Video_Generator" (id `1akzdENiWG3D4zu4zSLUv4B4xLrA98GO8-oDr1tA4uc4`), tab "Agents", one row per agent (`Email`, `Name`, `Phone`, `HeadshotUrl`, `Website`, `Instagram`), read-only lookup. `HeadshotUrl` is no longer used by the render (the realtor's photo now comes from a frontend upload instead) вЂ” the column can stay in the sheet unused or be removed, it doesn't matter.
- **n8n Data Table** вЂ” `ListingVideoJobs` (id `2j4dIhU0LEFKBFA8`): one row per submission (`submissionId`, `status`, `videoUrl`, `creatomateRenderId`, `agentEmail`, `errorMessage`, `pollAttempts`, `rawIntakePayload`), written and updated by n8n as each render progresses. Chosen over a second Google Sheets tab because it's written/updated multiple times per render and a native Data Table avoids Sheets' write-concurrency/rate-limit quirks.
- **Creatomate** вЂ” the render engine: Ken Burns zoom/pan, lower thirds, transitions, background music, final video assembly. One template, built visually, reused per render.
- **Vercel** вЂ” hosts the frontend intake form (`/frontend/`), independent deploys from the n8n workflow.

- **Two n8n webhooks/workflows**: `Listing Video Intake, Render & Poll` (starts the pipeline, returns a `submissionId` immediately) and `Listing Video Status Check` (reads the `ListingVideoJobs` Data Table row for a given `submissionId` and returns JSON). Both webhook URLs are stored as Vercel environment variables (`N8N_INTAKE_WEBHOOK_URL`, `N8N_STATUS_WEBHOOK_URL`) вЂ” never hardcoded in frontend code.
- **The `ListingVideoJobs` Data Table is the shared state** between the two systems вЂ” the intake workflow writes it, the status workflow reads it.
- **Delivery is frontend-only**: the realtor gets the video by staying on the submission page and clicking Download once it's ready вЂ” there's no email/Slack fallback. Since there's no way to recover the video if they navigate away, the frontend shows a persistent "Don't close this window вЂ” your video is being generated" message for the whole pending/rendering state.

## 7. Frontend (`/frontend/`)

**What it needs to do:**
- Collect the same intake fields the n8n workflow expects: 3 property photo uploads, **the realtor's own photo upload**, price, address, bed/bath, sqft, year built, key features, description, and the agent's email (used for the Sheets lookup вЂ” name/phone/socials only, not the photo). Key features and description are frontend-only вЂ” the Creatomate template has no slot for them.
- On submit, POST to the n8n intake webhook and receive a `submissionId`.
- Poll the n8n status webhook with that `submissionId`. While pending/rendering, show a persistent "Don't close this window вЂ” your video is being generated" message. Once complete, show a **Download** button (linking to the video URL). This is the only delivery mechanism вЂ” no email or Slack notification.
- Nothing else вЂ” no business logic, validation beyond basic form checks, or video processing happens client-side; that all stays in n8n + Creatomate.

**Where it lives:** `/frontend/` вЂ” a Next.js (App Router, TypeScript, Tailwind v4) app, deployed to Vercel independently of the n8n workflow. Built with the `frontend-design` skill for the visual/UX work вЂ” see В§8 for the design direction.

**Photo uploads в†’ Vercel Blob, not through the server:** the 4 photo files (3 property + realtor) are uploaded **directly from the browser** to Vercel Blob via `upload()` from `@vercel/blob/client`, authorized by a short-lived token minted by `app/api/blob-upload/route.ts`. This is deliberate, not incidental вЂ” routing real phone-camera photos (3вЂ“10MB each) through a Server Action or Route Handler risks Vercel's serverless body-size ceiling (Server Actions cap at 1MB by default; general function limits are ~4.5MB). Client-direct upload means file bytes never pass through a Next.js server function at all. Only after all 4 uploads resolve does the client call `submitIntake()` (a Server Action) with the resulting URLs.

**Conventions for the agent when generating or editing frontend code:**
- Keep `N8N_INTAKE_WEBHOOK_URL`, `N8N_STATUS_WEBHOOK_URL`, and `BLOB_READ_WRITE_TOKEN` in environment variables (`frontend/.env.local` locally, Vercel project settings in deployment) вЂ” never hardcode a webhook URL or token in committed code. `frontend/.env.example` documents all three.
- Prefer routing webhook calls through Next.js Route Handlers/Server Actions rather than calling n8n directly from client-side JS, so the webhook URLs aren't exposed in the browser bundle. The one exception is the Blob upload itself, which is intentionally client-direct (see above) вЂ” it doesn't touch the n8n webhooks.
- Don't duplicate pipeline logic (render triggering, status computation) in the frontend вЂ” it should only submit, poll, and display.
- The status proxy (`app/api/status/route.ts`) treats a 404 from the n8n status webhook as still-pending, not an error вЂ” the intake workflow responds with a `submissionId` in parallel with (not gated on) its Data Table row write, so a client can legitimately poll before that row exists.
- Use Vercel preview deployments to test form/UI changes before promoting to production.

## 8. Frontend Design Direction

The intake form and status page are deliberately styled as a two-act experience rather than a generic SaaS form:

- **Act I вЂ” the intake slip** (`/`): warm kraft-paper palette (`--paper`, `--paper-card`, `--ink`, `--stamp` red, `--brass`), a heavy poster slab serif (Alfa Slab One) for headlines, and a typewriter face (Special Elite) for labels/badges вЂ” evokes a real estate office's paper intake ticket, not an editorial cream-and-serif template. Signature element: a rotated rubber-stamp badge ("Listing Intake В· File No. ####") that settles into place on load.
- **Act II вЂ” the screening room** (`/status/[submissionId]`): dark near-black palette with an amber "projector" accent, same typewriter face for the status readout. Signature element: a 12-tick film-leader countdown ring that glows while pending, replaced by a "Now Showing" panel with a one-time light-sweep animation on completion.
- Both scenes respect `prefers-reduced-motion` and have visible keyboard focus rings in their own accent color.
- Design tokens live in `frontend/src/app/globals.css` (`@theme inline` block); fonts are loaded via `next/font/google` in `frontend/src/app/layout.tsx`.

## 10. Conventions for the Agent

- Never commit `.env`, or anything under `/temp/outputs/` or `/temp/resources/` вЂ” these are gitignored scratch/output space, not deliverables.
- n8n workflow JSON exports live in `/workflows/`. Once the Creatomate-only baseline is tested and working, tag/branch it (e.g. `v1-ken-burns`) before any future AI-animation upgrade is merged in.
- Treat the Creatomate template and the n8n workflow as the source of truth for pipeline behavior вЂ” `/tools/` scripts should be thin wrappers around them, not reimplementations.
- Check `/workflows/` for the relevant procedure file before making pipeline changes; update the procedure file if the actual workflow changes.
- This is a foundation doc for a small, deadline-driven build вЂ” keep additions simple and avoid introducing new services/tools without flagging the tradeoff first.
